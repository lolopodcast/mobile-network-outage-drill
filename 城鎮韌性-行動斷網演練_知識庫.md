# 城鎮韌性－行動斷網演練 知識架構（Knowledge Base）

---

## 1. 定義（Definitions）

### 行動斷網演練（Mobile Network Outage Drill）
- 模擬行動網路（4G/5G、行動數據、簡訊）全面或間歇中斷情境下，個人與家庭是否仍能維持基本生活機能、聯繫家人、確認安全的實作型演練。
- 回答問題：
  - 網路一旦斷了，我現在的生活習慣有哪些環節會立刻失效？
  - 這些失效環節，有沒有離線／實體的替代方案？
  - 這個替代方案，是「知道就好」還是「實際演練過確認可行」？

### 城鎮韌性（Community/Urban Resilience）
- 城鎮／社區在災害、基礎設施中斷等衝擊下，維持基本運作並快速恢復的能力，其中個人與家庭層級的斷網準備度是重要組成之一。
- 回答問題：
  - 個人準備行動如何疊加成社區層級的韌性？

## 2. 核心架構（Core Framework）

### 2.1 七大準備面向
1. 通訊備援與會合機制（含新增：實體會合點）
2. 空間與路線熟悉（避難處所）
3. 資料離線化
4. 現金與實體支付備援
5. 電力備援（新增）
6. 資訊查證素養
7. （社區層級）事後複盤與經驗回饋 <!-- 待確認：原方案未明列，屬建議延伸項，供使用者評估是否納入 -->

👉 核心邏輯：
- 每個面向對應「斷網情境下，日常生活中哪一項『看不見的網路依賴』會失效」
- 面向 1、5 是「聯繫得上人」的基礎；面向 2、3、4 是「聯繫不上人時，自己是否還能行動」的基礎；面向 6 是「訊息紊亂時，避免製造二次混亂」的基礎

### 2.2 演練流程
Input（平時準備：號碼、截圖、離線資料、現金、行動電源）
→ Process（實際執行：走一次避難路線、離線測試、無行動支付購物、模擬查證情境）
→ Output（找出「以為可行、實際不可行」的斷點，例如檔案需連雲端下載）
→ Value（修正備援方案，逐步累積成城鎮韌性）

## 3. 各項行動細節（Action Items Detail）

### 3.1 通訊備援與會合機制
- 事先確認家人、伴侶與重要同事的電話號碼
- 約定簡訊格式：「我平安／目前在＿＿／接下來前往＿＿／預計＿＿點再次聯絡」
- **新增：實體會合點與時間**——若完全斷訊、簡訊也送不出去，約定「無法聯絡時，幾點在哪裡碰面」作為保底機制

👉 關鍵概念：
- 約定簡訊格式 ≠ 斷網保底機制：簡訊格式的前提是網路「間歇性恢復」或能連上其他 Wi-Fi；若行動網路**完全**中斷，簡訊同樣送不出去
- 實體會合點 = 真正不依賴任何網路的備援方案，應與簡訊格式並列，而非被其取代

### 3.2 空間與路線熟悉
- 演練前查好最近避難處所位置並截圖保存（離線可查）
- 斷網期間不使用 Google Maps，實際走一次
- 確認事項：入口位置、步行所需時間、主要道路受阻時的替代路線

### 3.3 資料離線化檢查
檢查手機裡以下資料是否「真正離線可用」，而非「看起來存在但需連雲端下載」：
- 避難處所位置與離線地圖
- 證件及緊急聯絡資訊
- 醫療與用藥資料
- 車票、住宿地址與電子票券
- 工作文件及合作夥伴名單

👉 關鍵概念：
- 「檔案顯示在手機裡」≠「檔案已離線儲存」：雲端同步 App（相簿、雲端硬碟、票券 App）常只存縮圖或索引，內容仍需連網下載，斷網時才會發現打不開

### 3.4 現金與實體支付備援
- 準備少量現金、零錢、實體信用卡與悠遊卡
- 實際購買一樣東西，觀察沒有行動支付、叫車、外送、會員 App 時生活是否仍可運作

### 3.5 電力備援（新增）
- 出門前確認行動電源已充飽電
- 家中備妥備用電池或手搖／太陽能充電設備

👉 關鍵概念：
- 斷網情境常伴隨停電或基地台超載限流，手機沒電比沒訊號更早讓所有備援方案失效，屬於前置於通訊備援的基礎條件

### 3.6 資訊查證素養
- 收到資訊時先自問：
  - 這是官方通知、親眼所見，還是別人的轉述？
  - 現在真的需要立刻轉傳給其他人嗎？
- 無法查證時，先傳送「自己的位置、狀況與需求」，不急著散布未確認的消息

👉 關鍵概念：
- 查證優先於傳播：在通訊頻寬有限的斷網情境下，未經查證的轉傳會排擠真正需要傳遞的求助與安全訊息

## 4. 常見情境落差與修正（Gaps & Corrections）

| 原始假設 | 落差 | 修正方向 |
|---|---|---|
| 約定簡訊格式即可聯繫家人 | 情境是「行動斷網」，簡訊可能同樣送不出去 | 加註適用情境（間歇性恢復），並補「實體會合點與時間」作為完全斷訊時的備援 |
| 手機裡的資料可離線使用 | 許多檔案僅同步索引，需連雲端才能下載完整內容 | 演練時逐項打開確認，而非僅檢查「檔案是否顯示存在」 |
| 只準備通訊與資料備援 | 忽略斷網常伴隨的電力／停電風險 | 新增行動電源與備用電力檢查項目 |

## 5. 演練紀錄模板（Drill Log Template）— Level 4 起點

第 4 節的落差表格是**設計階段**依邏輯推論出來的落差（尚未實測）。這一節提供可重複使用的空白模板，讓每次**實際執行**演練後，把真實發現填進來，逐步把 KB 從「推論出的知識庫」升級為「驗證過、持續更新的活知識庫」。

### 5.1 使用方式
1. 演練前複製下方空白表格
2. 依 3.1～3.6 六大面向逐項實測，一項對應一列
3. 每次演練填一整批列，並標註演練日期，不覆蓋舊紀錄
4. 每季或每次演練後，檢視「累積出現 ≥ 2 次」的落差，優先排入修正行動

### 5.2 空白模板

| 演練日期 | 面向 | 檢查項目 | 預期結果 | 實際結果 | 是否有落差 | 落差說明 | 修正行動 | 修正期限 |
|---|---|---|---|---|---|---|---|---|
| | | | | | 是／否 | | | |

### 5.3 面向對照（填表時參考代號）
- 3.1 通訊備援與會合機制
- 3.2 空間與路線熟悉
- 3.3 資料離線化檢查
- 3.4 現金與實體支付備援
- 3.5 電力備援
- 3.6 資訊查證素養

👉 關鍵概念：
- 第4節＝**設計階段**推論落差；第5節＝**執行階段**實測落差，兩者應互相對照——設計階段推論的落差是否在實測中被驗證？有沒有設計時沒想到、但實測才發現的新落差？
- 累積數次演練紀錄後，第4節的表格內容應該用第5節的真實數據回頭更新，而不是永遠停留在推論版本
